# 任务执行流程规范

## 概述

本规则定义每次任务执行的标准流程，确保技能使用的合理性、修改的可追溯性，并持续优化技能体系。

---

## 阶段一：任务接收与技能评估

### 1.1 接收任务
收到用户任务后，首先：
- 理解任务目标
- 识别任务类型（开发/修复/重构/查询等）
- 评估任务复杂度

### 1.2 调用 skill-manager 评估技能
**必须执行**：在开始任何工作前，调用 `skill-manager` 技能进行技能评估。

评估内容：
1. 任务涉及的模块（server/web）
2. 需要的技能列表
3. 技能使用优先级
4. 技能间的协作关系

### 1.3 技能选择决策树

```
任务类型 → 推荐技能组合
─────────────────────────────────────
代码修改 → modification-boundary + git-commit
Bug 修复 → modification-boundary + git-commit
新功能 → skill-manager + modification-boundary + git-commit
代码查询 → skill-manager
Git 操作 → git-commit
技能管理 → skill-manager
复杂任务 → skill-manager + modification-boundary + git-commit
```

### 1.4 技能评估输出格式

```markdown
## 技能评估

**任务类型**: <type>
**涉及模块**: <modules>

**推荐技能**:
| 技能 | 使用原因 | 优先级 |
|-----|---------|-------|
| skill-manager | 协调技能加载 | 高 |
| modification-boundary | 限制修改范围 | 高 |
| git-commit | 提交代码变更 | 中 |

**技能协作流程**:
<描述技能如何协同工作>
```

---

## 阶段二：任务执行

### 2.1 加载技能
根据评估结果，使用 `skill` 工具加载所需技能：
```
skill({ name: "skill-name" })
```

### 2.2 遵守修改边界
如果涉及代码修改：
- 声明允许修改的模块
- 声明禁止修改的模块
- 声明只读参考的模块
- 严格遵守边界执行修改

### 2.3 执行记录
执行过程中记录：
- 访问的文件列表
- 修改的文件列表
- 使用的技能
- 关键决策点

### 3.2 报告填写规范

**修改类型**:
- `新增`: 新创建的文件
- `修改`: 已存在文件的修改
- `删除`: 删除的文件
- `重命名`: 文件重命名

**修改内容摘要**:
- 简洁描述主要变更
- 不超过 50 字

**修改原因**:
- 说明修改的业务/技术原因
- 关联需求或问题

**技能使用原因**:
- 说明为什么选择该技能
- 该技能解决了什么问题

## 阶段四： Git 提交（如适用）

### 4.1 提交检查
如果任务涉及代码修改，执行：
1. `git status` 检查变更
2. `git diff` 预览变更
3. 确认变更范围正确

### 4.2 提交执行
调用 `git-commit` 技能进行提交：
- 生成规范的提交信息
- 遵循 Conventional Commits 规范
- 验证提交成功

### 4.3 提交报告
在结果报告中包含：
```markdown
### Git 提交信息

**提交 Hash**: <short-hash>
**提交信息**: <commit message>
**提交时间**: <timestamp>
```

## 特殊情况处理

### 情况 1: 纯查询任务
无需代码修改的任务：
- 跳过修改边界声明
- 跳过 Git 提交
- 仍需输出技能使用情况

### 情况 2: 技能评估不需要某些技能
如果评估后认为某些技能不必要：
- 在技能使用情况中标记为"否"
- 说明不使用的理由

### 情况 3: 任务执行中发现新需求
如需调整技能或修改范围：
- 重新调用 skill-manager 评估
- 更新修改边界声明
- 在报告中说明调整原因

### 情况 4: 任务失败或中断
- 记录失败原因
- 记录已执行步骤
- 给出恢复建议

---

## 技能体系持续优化

### 技能使用统计
定期分析：
- 哪些技能使用频率高
- 哪些技能从未使用
- 哪些场景缺少技能支持

### 新技能触发条件
当出现以下情况时，建议创建新技能：
1. 重复性任务出现 3 次以上
2. 现有技能无法覆盖的场景
3. 需要专业化处理的领域
4. 团队协作需要标准化流程

### 技能迭代
- 根据使用情况更新技能描述
- 补充新的使用场景
- 优化工作流程

## 示例报告

```markdown
## 任务执行报告

### 修改文件汇总

| 文件路径 | 修改类型 | 修改内容摘要 | 修改原因 |
|---------|---------|-------------|---------|
| content-moderation-server/src/main/java/com/midust/parser/PromptResolver.java | 修改 | 添加默认值回退逻辑 | 修复空指针异常 |
| content-moderation-server/src/main/java/com/midust/parser/PromptTemplates.java | 修改 | 更新 Prompt 模板内容 | 支持新违规类型 |

### 详细修改说明

#### 文件 1: PromptResolver.java
**修改类型**: 修改

**修改内容**:
```java
// 新增回退逻辑
String safeLocale = StringUtils.defaultIfBlank(locale, parserProperties.getDefaultLocale());
```

**修改原因**:
当传入的 locale 为空时，使用默认 locale 回退，避免空指针异常。

---

### 技能使用情况

| 技能 | 是否使用 | 使用原因 | 使用时机 |
|-----|---------|---------|---------|
| skill-manager | 是 | 评估任务需要的技能 | 任务开始前 |
| modification-boundary | 是 | 确保只修改 parser 模块 | 代码修改前 |
| git-commit | 是 | 提交代码变更 | 任务完成后 |

**技能使用详情**:

**skill-manager**:
- 使用原因：识别这是后端修复任务，需要修改 parser 模块
- 调用时机：任务开始前
- 输出结果：推荐使用 modification-boundary 和 git-commit

**modification-boundary**:
- 使用原因：确保只修改 parser 模块，不影响其他模块
- 修改边界声明：
  - 允许修改：content-moderation-server/src/main/java/com/midust/parser/**
  - 禁止修改：content-moderation-server/src/main/java/com/midust/planner/**
  - 只读参考：content-moderation-server/src/main/java/com/midust/model/**
- 验证结果：遵守边界，只修改了 parser 模块

**git-commit**:
- 使用原因：完成代码修复后提交
- 提交信息：fix(parser): 修复 Prompt 解析空指针异常
- 提交状态：成功

---

### 新技能建议

**是否需要新技能**: 否

本次任务现有技能已满足需求，无需新技能。

---

### 后续建议

建议添加单元测试覆盖 PromptResolver 的回退逻辑。
```

## 规则总结

| 阶段 | 必须执行 | 输出内容 |
|-----|---------|---------|
| 技能评估 | ✅ | 技能选择决策 |
| 任务执行 | ✅ | 代码变更 |
| 结果报告 | ✅ | 修改文件 + 技能使用 + 新技能建议 |
| Git 提交 | 条件 | 提交信息 |

**核心原则**:
1. 先评估，后执行
2. 修改必有边界
3. 完成必有报告
4. 持续优化技能
