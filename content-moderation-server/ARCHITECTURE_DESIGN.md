# Content Moderation Server 架构设计文档

## 1. 架构愿景

基于架构图设计，构建一个**企业级 AI 内容风控平台**，支持多场景、多模态、多模型的内容审核与风控能力。

### Phase 3: 智能规划（v3.0 - 4~6 个月）

**目标**：引入 Planner 和 Critic 模型

**核心能力**：
```
┌─────────────────────────────────────────┐
│          智能规划层                      │
│  ┌───────────┐  ┌───────────┐          │
│  │ 任务拆解  │  │ 智能 Prompt│          │
│  │ Planner   │  │ 生成器    │          │
│  └───────────┘  └───────────┘          │
│  ┌───────────┐  ┌───────────┐          │
│  │ 验证器    │  │ 上下文管理│          │
│  │ Validator │  │           │          │
│  └───────────┘  └───────────┘          │
└─────────────────────────────────────────┘
```

**Planner 模型职责**：
1. 接收复杂审核任务
2. 拆解为多个子任务（如：先图片→再文本→最后视频）
3. 调度 Executor 模型执行
4. 汇总结果

**Critic 模型职责**：
1. 对 Executor 输出进行质量评估
2. 置信度校验
3. 触发复审机制

**示例流程**：
```
复杂审核任务
    ↓
Planner 模型
    ↓
┌─────────────────────────────────┐
│ 子任务 1: 图片审核 (Executor)   │
│ 子任务 2: 文本审核 (Executor)   │
│ 子任务 3: 视频审核 (Executor)   │
└─────────────────────────────────┘
    ↓
Critic 模型（质量评估）
    ↓
最终审核结果
```

### Phase 4: 反馈学习（v4.0 - 6~9 个月）

**目标**：建立策略优化闭环

**核心组件**：
```
┌─────────────────────────────────────────┐
│          反馈学习层                      │
│  ┌───────────┐  ┌───────────┐          │
│  │ 策略路由  │  │ 任务状态机│          │
│  │ Policy    │  │ Task State│          │
│  │ Engine    │  │ Machine   │          │
│  └───────────┘  └───────────┘          │
│  ┌───────────┐                         │
│  │ 失败重规划│                         │
│  └───────────┘                         │
└─────────────────────────────────────────┘
```

**策略路由（Policy Engine）**：
- 根据内容类型、风险等级、历史数据，动态选择审核策略
- 示例：高风险用户 → 严格策略，低风险用户 → 快速策略

**任务状态机**：
```
PENDING → PROCESSING → VALIDATING → COMPLETED
              ↓              ↓
         RETRY(3)      MANUAL_REVIEW
```

**失败重规划**：
- Executor 失败 → 重试或切换模型
- Critic 不通过 → 重新执行
- 超过阈值 → 转人工审核

### Phase 4.1 Policy 执行流程（当前落地流程）

为统一 Skill 与 Policy 的执行语义，执行链路按以下步骤推进：

1. 加载 Policy
   - 根据 `policyId` 读取策略定义
   - 获取 `skillPipeline` 与 `config`

2. 组合 Skill Pipeline
   - 按 Policy 中配置顺序构建待执行 Skill 列表
   - 结合 `config` 完成开关、阈值、跳过规则预处理

3. Planner 生成 SKILL Plan
   - 以 `skillPipeline + 输入上下文 + config` 作为 Planner 入参
   - 输出可执行计划（步骤、依赖、跳过条件、重试策略）

4. 执行 Plan
   - 按计划顺序执行每个 Skill
   - 单个 Skill 执行结果记录 `success / skipped / duration / message / output`
   - 失败时按计划策略重试、降级或中断

5. 写入 State
   - 初始化写入 `input`
   - 每个 Skill 成功后写入 `state[skillId] = output`
   - 结束后写入 `policyId / success / durationMs / errorMessage`

执行主流程可抽象为：

```
加载 Policy
    ↓
组合 skillPipeline
    ↓
Planner 生成 SKILL Plan
    ↓
执行 Plan（按步骤执行）
    ↓
写入 state（input + 每步output + 汇总元信息）
```

约束说明：
- Skill 执行模式统一为 LLM 路径，Policy 层不再依赖 Java 执行器分流
- Skill 在注册阶段完成 LLM 绑定，执行阶段按 Skill 绑定配置进行模型调用
- Policy `config` 仅作为策略参数，不直接承载执行结果

## 5. 技术债务与重构计划

### 5.1 当前技术债务

| 问题 | 影响 | 优先级 |
|------|------|--------|
| 硬编码 LLM 配置 | 难以切换模型 | 高 |
| 缺少模型抽象 | 无法支持多模型 | 高 |
| 无任务状态管理 | 失败后无法追踪 | 中 |
| 无统一错误处理 | 错误信息不一致 | 中 |
| 缺少监控指标 | 无法评估性能 | 低 |

## 6. 关键设计决策

### 6.1 为什么分层？

**问题**：为什么不把所有功能放在一个服务里？

**答案**：
- **关注点分离**：每层只负责一件事
- **独立演进**：可以单独升级某一层
- **团队协作**：不同团队负责不同层
- **故障隔离**：一层故障不影响其他层

### 6.2 为什么需要 Planner 模型？

**问题**：直接调用 Executor 不行吗？

**答案**：
- **复杂任务处理**：一个任务可能需要多次审核
- **成本优化**：简单任务用便宜模型，复杂任务用贵模型
- **质量控制**：Planner 可以决定是否需要 Critic 复审

### 6.3 为什么需要反馈学习？

**问题**：模型不是已经训练好了吗？

**答案**：
- **业务变化**：违规类型会随时间变化
- **模型退化**：模型性能会随时间下降
- **策略优化**：需要根据实际效果调整策略
