## 一、系统定位与核心价值

### 1.1 系统本质

**这不是一个传统的"风控系统"，而是一个面向交通与全球化领域的「智能视频内容管理与服务质量分析平台」。**

核心业务场景：
- **全球化支持**：覆盖 LinkU、海外华人等多元化业务场景。
- **录屏会话数据采集**：含音频轨道与视频帧，遵循**标准执行协议层**（通用文本/图片/视频/语音）。
- **服务质量分析流水线**：基于 AI Agent 架构的自动化分析。
- **多模态语义解析与要素抽取**：解构复杂上下文，而非简单的单句风控。
- **服务过程健康度评估**：从合规、态度、风格多维度刻画。

### 1.2 与"风控"的关系

**风控只是系统的副产品**，真正的核心价值在于：

| 价值层级 | 占比 | 说明 |
|---------|------|------|
| **智能规划** | 50% | 动态调度能力、任务拆解、反馈学习闭环 |
| **数据仓库** | 30% | 证据融合、标签仓库、用户/主播画像 |
| **服务质量评估** | 15% | 服务态度、交互风格、健康度评估 |
| **风险检测** | 5% | 隐性导流、人设矛盾等深度风险识别 |

## 三、AI 核心架构：Planner-Executor-Critic

系统采用先进的 AI Agent 架构，确保复杂任务的高效执行与自我进化。

### 3.1 角色定义
- **Planner (规划者)**：
    - **职责**：根据输入意图、复杂度和约束条件，动态决定处理策略、任务拆解方式及 Prompt 模板。
    - **能力**：初始化、结构化输入管理、任务拆解 (Planner)、上下文管理。
- **Executor (执行者)**：
    - **职责**：执行具体的原子任务（如 ASR、CV 识别、语义抽取）。
    - **能力**：豆包图文理解、视频理解、垂直领域模型、通用大模型。
- **Critic (评估者)**：
    - **职责**：对 Executor 输出的结果进行结构、规则和业务逻辑校验。
    - **能力**：验证器 (Validator)、结果评分、反馈学习机制。

### 3.2 反馈学习闭环 (Human-in-the-loop)
系统构建了 **Planner -> Execution -> Evaluation -> Policy Optimization** 的闭环：
1. **结果评估**：对输出结果进行效能评分或人工反馈修正。
2. **策略优化**：根据评分反向调整调度策略、Prompt 模板或任务拆解方式。
3. **失败重规划**：任务状态机监控执行过程，遇失败自动触发重规划机制。

## 四、核心业务概念

### 4.1 会话 (Session)
**定义**：一次完整的录屏会话，遵循标准执行协议，包含多模态数据。

### 4.2 服务质量画像 (Service Quality Profile)
**定义**：基于**语义向量**与**多源证据融合**的综合画像，支持跨场景对比。

### 4.3 治理级标签 (Governance Tag)
**定义**：稳定、可解释、可复用的标签，作为系统最终产出物。

## 五、智能规划引擎深度解析

### 5.1 核心组件
- **任务状态机 (Task State Machine)**：用一组明确的状态及其流转规则，管理任务从创建到完成的全过程执行与重规划。
- **验证器 (Validator)**：LLM 输出后，进行 Schema 与约束校验，不合格则触发修正或重试。
- **语义向量 (Semantic Vector)**：将文本、标签、行为转化为可计算的表示，支持相似度检索、用户画像匹配等。
- **策略路由 (Policy Engine)**：动态决策后续采用哪种处理路径，实现成本与效果的最优平衡。

### 5.2 调度示例
```typescript
// 场景：海外华人高价值用户投诉件
if (session.context.scene === 'OverseasChinese' && user.priority === 'High') {
  plan.planner.strategy = 'EnhancedReasoning'; // 启用增强推理策略
  plan.executor.models.push('MultiModal_Expert'); // 调度多模态专家模型
  plan.critic.validationLevel = 'Strict'; // 启用严格校验
}
```

## 十一、Prompt 可拔插化配置方案（新增）

### 11.1 目标
- 将“单一超长 Prompt”升级为“模块化 Prompt 组合”。
- 支持前端勾选分析模块，后端按规则拼接最终 Prompt。
- 支持版本化与任务追溯，确保线上行为可复现。

### 11.2 模块拆分建议
- `BASE_TASK`：任务说明、角色范围、输出原则。
- `RULE_13_TYPES`：13 类违规判定规则主块。
- `ANTI_FALSE_POSITIVE`：误判防范规则。
- `JSON_OUTPUT_SCHEMA`：严格 JSON 输出约束。
- `HEALTH_SCORE_ADJUSTMENT`：健康分扣减建议块。
- `ROLE_JUDGE_RULE`：男女角色判断规则。
- `BACKGROUND_CONTINUITY_RULE`：背景连续性与场景切换规则。
- `CALL_IN_BED_STRICT_RULE`：在床通话严格判定块（可单独开关）。
- `BLACK_SCREEN_STRICT_RULE`：黑屏与无人出镜边界规则。

### 11.3 后端实现路径
1. 新增 `PromptModule` 配置模型，包含 `code/name/content/version/enabled/order`。
2. 新增 `PromptComposer` 组件，按 `selectedModuleCodes` 与 `order` 拼接。
3. 将 `LLMIntegrationServiceImpl.buildPrompt` 改为调用 `PromptComposer`。
4. 在任务分析请求中增加 `promptModules` 字段。
5. 持久化“模块列表 + 最终 Prompt 快照 + 版本号”用于审计与回放。

### 11.4 前端实现路径
1. 在“发起分析”页面提供模块勾选面板（含默认推荐组合）。
2. 支持“保存为模板组合”（例如：严格风控版、体验评估版）。
3. 提交任务时将选中的模块编码传给后端。
4. 在任务详情页展示本次使用的 Prompt 模块与版本信息。

### 11.5 配置示意（逻辑）
- 配置文件可采用 `prompt-modules.yml`，每个模块独立管理。
- 支持灰度开关：按业务线、国家站点、用户分层切换模块组合。
- 支持 A/B 对比：同一视频使用不同模块组合评估效果差异。

### 11.6 收益
- 新增规则无需改核心代码，仅新增模块并勾选即可生效。
- 提示词维护成本显著降低，规则可复用、可组合、可版本化。
- 便于快速迭代与线上问题追踪。

## 结语

本系统的核心竞争力在于**智能规划 (Planning)** 与 **反馈进化 (Feedback)**。通过构建 Planner-Executor-Critic 的 Agent 架构，系统不仅能高效处理多模态数据，更能随着业务反馈不断自我优化，从单一的分析工具进化为智能治理平台。
